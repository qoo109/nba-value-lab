# NBA Value Lab V2.6：爬蟲與儲存規格

## 原則

V2.6 不使用 API 金鑰，也不建立「10 家莊家、賽前 24 小時每 5 分鐘」的全量資料庫。資料擷取僅能使用完成來源審查的爬蟲或核准檔案匯入。

公開可讀不代表允許自動收集、長期保存或再發布。每個來源啟用前必須保存：

- 來源名稱與 URL
- robots 檢查結果
- 服務條款與授權判斷
- 可接受請求頻率
- 可保存欄位與期限
- 審查日期與負責版本

## 稀疏快照

每場只保存以下時間：

1. Opening：市場賠率起點
2. T−24h：提前預覽
3. T−6h：可選漂移檢查
4. T−60m：主要模型與分級鎖定
5. T−5m：傷病、先發與市場賠率複核
6. Closing：只計算 CLV，不回改原預測

若發生核心球員狀態、先發、比賽時間或市場賠率的重大變化，可新增 `event` 快照，但必須保存觸發原因。

## 來源數量

- 目標莊家：1 家
- 比較來源：3～5 家
- 雙方賠率必須來自同一莊家、同一市場、同一時間快照
- 每家先獨立去水，再形成市場共識
- 市場共識應排除使用者的目標莊家

## 每筆盤口最低欄位

- `game_id`
- `bookmaker_id`
- `market_id`
- `snapshot_label`
- `favorite_odds_decimal`
- `underdog_odds_decimal`
- `source_url`
- `published_at`
- `observed_at`
- `fetched_at`
- `content_hash`
- `parser_version`
- `settlement_rule`
- `includes_overtime`

## 容量估算

以四季、每季約 1,230 場、4～6 個來源、5～6 個快照估算，盤口主表約 15～20 萬筆。若保存 10 家莊家、24 小時每 5 分鐘，可能接近 1,530 萬筆，因此 V2.6 明確停用全量模式。

## 儲存分工

GitHub 只保存：

- 程式碼
- 模型文件
- JSON Schema
- migration
- 小型合成 fixture

GitHub 不保存：

- 正式歷史資料庫
- 大量原始 HTML/PDF
- 完整市場賠率快照
- 含個資、憑證或未確認再利用權的資料

正式資料應放在可備份、可版本化且權限受控的資料庫或物件儲存，並依月份或賽季匯出 Parquet 封存。
